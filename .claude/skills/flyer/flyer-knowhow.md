## 1. ワークフロー：作って終わりにしない

```
生成スクリプト実行 → PNG書き出し → 自分の目で見る → 修正 → 再生成 → …収束するまで
```

- **コードが正しくてもデザインは壊れている**。テキストの折返し・要素の重なり・z-order逆転は、レンダリングして見るまで絶対に分からない。
- PNG書き出しは PowerShell + PowerPoint COM（`export-slides.ps1`）で行う。

```
powershell -File export-slides.ps1 -PptxPath "flyers\<商品名>\xxx.pptx" -OutDir "flyers\<商品名>\_preview"
# 納品JPG: -Format JPG -Names "案名1","案名2",...
```

- **LibreOfficeでのプレビューは使わない**：合成太字レンダリングで字形が崩壊する（Shippori Mincho Bold の横書き・高密度スライドで「の本格スープ」が潰れた実例）、高密度スライドでXMLが正しくてもプレビューが不正確になる、という2点でQAに使えないと判断済み。PowerPoint COM に統一する。
- **ユーザーに見せるのは「インパクト・創意性まで仕上がってから」**。技術的に破綻していないだけの状態で見せない。
- 問題の原因が分からない時は**ビセクション**（要素を段階的に分離して原因特定）。

### 目視QAチェックリスト（毎サイクル確認）
1. テキストの**意図しない折返し**（「！」「。」だけ2行目に落ちる等）→ `wrap: false` で対策
2. 要素同士の**重なり・衝突**（価格と擬音、バッジと画像、瓶ラベル同士、写真と情報カード）
   → 重なりは y座標の垂直オフセットを数値管理で広げて解消（例: 要素A y≈4.95 / 要素B y≈7.6 のように記録しながら調整）
3. **z-order**：後から `add` したものが前面。バッジを画像より先に描くと隠れて消える
4. **フレーム系素材との干渉**：枠画像の縁は思ったより太い。縁を実測してから内側に安全域を設定
5. **断ち落とし**のつもりが「見切れ」になっていないか（重要文字が端で切れるのはNG）
6. 縦書きテキスト列の**カラム折り返し**：バンド高さを十分確保する
7. 縮小表示（サムネイル）で見て、**一番伝えたい情報が最初に目に入るか**

## 2. レイアウト原則

- **視線誘導はZパターン**：左上（キャッチ）→右上→中央（メインビジュアル）→左下→右下（行動情報）。
- **「上に写真／中にキャッチ／下に情報」の凡庸な三段構成に陥らない**。斜め配置・断ち落とし・特大タイポ・非対称で構図に動きを出す。
- **重心を決める**：主役（商品・手元・顔）を「画面全体の中心」に数式で合わせる。
  画像内の注目点の位置比率（focusRatio）から逆算する：
  `imgX = slideCenterX - boxW * focusXRatio`
- **情報量は極限まで削る**。商品名・価格・買える場所or期間、以上。書きたいことはコピー1本に集約。
- 要素を**少し回転（±3〜8°）**させるだけで勢いが出る。全部水平は死ぬ。
- 情報の再構成はユーモアと独創性の源泉：**テーマの文法に情報を翻訳する**（メニュー→公約、日付→投票日、価格→チーズの穴の中、等）。

## 3. 配色

- **3色まで**（メイン・サブ・アクセント）＋比率 **70:25:5**。
- 商品写真から主役色を1つ抽出し、それを軸に組む。固定パレットを使い回さない。
- 食欲を刺激するのは**赤・黄・オレンジの暖色**。食品は暖色を必ず一点。冷菓・清涼系は寒色併用OK。
- 「安っぽい」と感じたら原因は大抵 **フラットな単色塗り＋多色混在**。色数を絞って質感（写真・実測ベースの枠）を足す。
- 実績パレット（参考。コピペではなく発想の種に）：
  - 和スイーツ高級系: 小豆`96514D`×藍`165E83`×鳥の子`FFF1CF`×金`C9A227`
  - ダイナー: 赤`D42B2B`×黒`1A1A1A`×ターコイズ`40E0D0`
  - ネオン: 黒地`0A0012`×シアン`00F5FF`×マゼンタ`FF2D78`×ライム`CCFF00`
  - POP特価: ビビッド赤`D32F2F`×黄`FFEB3B`×黒`212121`（インパクト最強クラス）
  - チーズ/カスタード系: `FFE5A3`×`FCDB66`×`F19A3A`×`A8785D`
  - 選挙風: 白×紺`1B3F8F`×赤`D7000F`

## 4. タイポグラフィ

- **太字にはクッキリした「フチ」**（オフセット複製8方向）。**ぼかし影とフチを重ねると安っぽくなる**——どちらか一方。
- 価格・訴求ポイントは**恥ずかしいくらい大きく**（本文の3倍以上）。強弱をつける（売りたい方を大きく）。
- オノマトペ（じゅわっ・とろ〜り・ぱりっ）は**独立したビジュアル要素**として斜めに大きく置く。引き出し線でつなぐと台無し。
- 手書きフォントは活字とのコントラストで効く。効かせどころを絞る（商品名・サイン・しずる文）。1案で活字2〜3種＋装飾フォント程度。
- 名前・見出しの**ひらがな化**は親しみやすさの definitive な手法。

## 5. 写真の扱い

- **透過PNG（removebg等）は必ずアルファ値bboxでクロップしてから使う**。透明余白込みでcontain-fitすると被写体が意図せず縮む（実測：最悪48%しか写っていなかった）。
- 写真の構図とボックスの縦横比を合わせる。合っていないと被写体が変な位置に浮く。
- **顔・商品にテキストを重ねない**。視線・ジェスチャー・盛り付けの方向に余白／文字を置く。
- 写真が持っている物語・スタイリングを核にテーマを立てると強い（§8）。
- **白背景の写真**を地色に馴染ませる3手：①地色と同色fill＋金線の矩形を下に敷いて「額装」②白カードに乗せる③地色を半透明で被せて沈める。
- **写真は非破壊が原則**：円形cover-cropは被写体が切れて不評。円形の台紙（ellipse）の上に透過PNGをそのまま重ねる方が「ステッカー感」も出て強い。

## 6. pptxgenjs 技術Tips（ハマりどころ）

| 現象 | 原因と対策 |
|---|---|
| PowerPointが「ファイル破損」と言う | ①シェイプの `w`/`h` が**負数**（線の座標計算は `Math.min`/`Math.abs` で正規化必須）②**`shadow: { type: 'inner' }`** — pptxgenjsが壊れたXMLを吐く。**inner shadowは使用禁止** |
| どのスライドが破損源か分からない | **二分探索**：スライドブロックを1枚ずつ個別pptxに書き出し、PowerShell COMでループopenして特定する（実績あり・数分で特定できる） |
| テキストが勝手に折り返す・箱からはみ出す | spAutoFitが効かない。`wrap: false` + 十分な `w`、大フォント複数行は行数×サイズから箱を逆算 |
| 縦書き（`vert: 'eaVert'`）が蛇行する | 促音「っ」の処理が不安定。**1文字ずつ個別テキストボックス**で縦に並べる（促音は0.75倍サイズ、句読点「、。」はマス高0.5倍で`align:'right'/valign:'top'`＝縦組の右上寄せ、長音「ー〜」は`rotate:90`）。実装は hotdog-stall の `generate-nine.js` の `vText()` を移植して再利用 |
| `sizing:{type:'cover'}`が写真を引き伸ばす | pptxgenjsのcover/containは**実画像サイズを読まずボックスサイズをそのまま「画像サイズ」として計算する**バグ。ボックスと画像のアスペクト比が離れているほど露骨に伸びる。実寸を取得し、top-levelの`w`/`h`に実アスペクト比を渡してから`sizing.w/h`でボックスを指定する`addCover()`ヘルパー（hotdog-stall `generate-nine.js`）を必ず使う |
| 画像を複数枚挿入するとバグりやすい | 1枚ずつ検証しながら実装する（一括で書いてまとめてデバッグしない） |
| 集中線がショボい/引き出し線に見える | 中心から生やすのではなく**四辺の等間隔点→焦点への収束線**で引く。太さ・透明度を2〜3段階混ぜる。**三角形の大量重ねは禁止**（線シェイプ十数本で表現） |
| 星形バースト(star16等)が安っぽい | プリセット星形のフラット塗りはクリップアート感。使わないか写真素材で代替 |
| 枠画像が中身を覆う | 枠の透明窓のbboxをpngjsで実測し、**窓がコンテンツ領域に一致する非等倍ストレッチ**を数式で計算（下記） |
| 生成できない（EBUSY） | PowerPointがファイルを開いている。**§7 のルールに従い、必ずユーザーに確認**。無断killは絶対禁止 |

### 枠画像フィットの計算式
```js
// fL,fR,fT,fB = 窓のbbox比率（実測）、border = 見せたい縁の太さ(in)
const boxW = (W - 2*border) / (fR - fL);
const boxH = (H - 2*border) / (fB - fT);
const boxX = border - fL * boxW;
const boxY = border - fT * boxH;
```

### 基本規約（生成スクリプト共通）
- A4縦：`defineLayout({name:'A4',width:8.27,height:11.69})`
- `hex` に `#` を付けない。8桁hex禁止（透明度は `transparency`/`opacity`）
- 円形抜きは正方形領域を `cover` → `rounding:true`（非正方形だと楕円）
- オプションオブジェクト（shadow等）は使い回さず毎回 `{}` 新規生成（共有参照バグ）
- 角丸＋枠の併用は `RECTANGLE`。装飾下線は使わない
- 断ち落とし（座標負値・W/H超）は意図的に使ってよい
- スライド番号と生成ブロックの対応をスクリプト冒頭コメントに必ず書く

## 7. 手動編集保護（例外なし・最重要）

**一度ユーザーに見せた pptx は手動編集されている前提で扱う。**

### 事故の記録（なぜこのルールがあるか）
姉妹プロジェクト hotdog-stall で、PowerPoint がファイルを開いていた際に「自分が直前にエクスポートした残骸だろう」と推測して無断で Stop-Process → 再生成し、**ユーザーの手動編集（いいね・コメントアイコンの手直し）を握りつぶした**。「同一セッション内の修正指示中なら確認なしで閉じてよい」という例外を設けていたのが原因。**例外は全廃した。二度とやるな。**

### 手順（再生成・上書きの前に必ず）
1. **ハッシュ照合**：`powershell -File scripts/check-flyer-hash.ps1 -File "flyers\<商品名>\xxx.pptx"`
   - `MODIFIED` → 手動編集あり。**絶対に上書きしない**。ユーザーに確認する
   - `NO_RECORD` → 承認記録なし。ユーザーに渡したことがあるファイルなら口頭で確認する
   - `MATCH` / `NO_FILE` → 再生成してよい（承認版は git にコミット済み）
2. **EBUSY / ファイルロック** = ユーザーが今開いているシグナル。理由を決めつけず、
   **「PowerPointが開いていて再生成できません。編集中でしたら保存してから教えてください」と聞く。それだけで防げる。**
3. **承認時**：`powershell -File scripts/approve-flyer.ps1 -Files "flyers\<商品名>\xxx.pptx"` でハッシュ記録 → `git add` + `git commit` でスナップショット化 → JPG書き出し
4. 手動編集を新スクリプトに取り込むときは、**元スクリプトではなく編集後の状態**（XML解析 or 確定JPG）から座標・サイズを再現する。元コードからコピーすると編集が消える。
5. 確定版のJPGは正解のスナップショット。迷ったら見比べる。

## 7.6 実戦で効いた「勝ちパターン」（ユーザー高評価の実績）

- **特大一文字**（「旨」320pt・透かし気味）＝ 実測で最強クラスのインパクト。1文字に意味を凝縮して画面の核にする。朱印と組むとさらに締まる
- **持ちポーズ×商品オーバーレイ**：手で輪を作る/構える写真に商品画像を重ねて「持たせる」と一気に物語が生まれる。位置は実測で合わせ、商品は小さめ＋軽い回転
- **モチーフへの情報埋め込み**（チーズの穴に価格・公約にメニュー）は毎回高評価。装飾を足すよりも**情報の置き場所を発明**する
- **ロゴの図形再現**：商品ロゴ（円形エンブレム等）を OVAL+テキストで再現し全案の軸にすると、ブランド感が跳ねる（冷やしぜんざい「ぜ」エンブレムの実績）
- **手書き書体×しずるフォト中心**への再構築で高級感デザインが化けた（同上）
- **写真は非破壊**：円形の台紙の上に透過PNG（§5）

## 7.7 素材参照とスクリプト編集の罠

- **`imgs[4]`のようなインデックス参照は禁止**。素材フォルダのファイル増減で全インデックスが黙ってずれ、複数スライドの写真が一斉に入れ替わる事故が起きた（実際に起きた）。`imageByName('main-sizzle')`のような**ファイル名解決**を使う
- **テキストの装飾記号（✦♥☺等）はフォント非対応だと黒い棒に化ける**。スパークルは`star4`シェイプ、ハートは図形で。★♪は和文フォントにあるので安全
- node -e をbashのダブルクォートで包んで置換スクリプトを書くとき、**テンプレートリテラルのバッククォートがシェルのコマンド置換として実行されて中身が消える**。複雑な置換はtmpファイル＋Editツールで。置換後は必ずgrepで着地確認（silent failする）
- 加工済みフォルダ（cropped等）がある場合は**加工済みフォルダを直接scanする**（元フォルダをscanして名前一致で差し替える方式は、加工済みにしか無い画像がリストから漏れて枠が空白になる。実際に発生）

## 8. テーマ発想のフレームワーク

1. **既存テーマと軸をずらす**：色（暖/寒）、時代（レトロ/未来）、文脈（食品広告/選挙/映画/コミック/号外/手紙/処方箋）
2. **持っている写真・商品の物語から逆算**（§5。スタイリング・ポーズ・器・産地が物語を持っている）
3. **「情報の言い換え」が成立するテーマは強い**（公約・指名手配・号外・処方箋・投票日…）
4. リサーチしてから作る：そのジャンルの**定石（記号）を3つ以上**入れると「それっぽさ」が出る
   （選挙なら：ひらがな名・たすき・公約・党名バッジ。ダイナーなら：チェッカー・ネオンサイン）
5. ギフト文脈の翻訳先も強い：**のし・熨斗紙・お歳暮カタログ・ラブレター・仕送り・お裾分け**など「贈る行為」の記号
